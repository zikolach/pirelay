## Context

Pi user messages can include `ImageContent` blocks (`{ type: "image", data, mimeType }`) through the extension API, and Pi session/tool-result messages can contain image blocks. The Telegram tunnel currently parses only `message.text` from Telegram updates and only sends text or Markdown documents back to Telegram. This leaves a gap for common mobile workflows such as sending a screenshot to the active Pi session or retrieving an image artifact generated by a tool.

Telegram Bot API image handling has several constraints: photos arrive as multiple compressed sizes, image documents preserve the original file but can be large, bots must call `getFile` and download bytes before forwarding, and the Bot API transport is not end-to-end encrypted. The tunnel must therefore authorize the chat/user before downloading files, apply strict file limits, and avoid automatically exfiltrating local images back to Telegram.

## Goals / Non-Goals

**Goals:**

- Support authorized Telegram photos and image documents as Pi image prompt inputs.
- Preserve captions and command semantics for image messages, including `/steer` and `/followup` captions.
- Reject unsupported or oversized images with actionable Telegram feedback.
- Detect image outputs from the latest Pi turn and expose them through an explicit Telegram retrieval action.
- Keep in-process and broker runtimes behaviorally equivalent.
- Keep image state ephemeral and bounded so session exports do not contain new secrets or large binary blobs beyond normal Pi message history.

**Non-Goals:**

- Supporting arbitrary non-image Telegram attachments.
- Automatically forwarding local user-attached images back to Telegram.
- Automatically sending all generated images to Telegram without explicit user action.
- Building a general file browser or workspace file download feature.
- Adding image resizing/transcoding as part of the first implementation unless needed for safe rejection fallback.

## Decisions

### Represent Telegram image input as Pi `ImageContent`

Extend inbound Telegram message types to include supported image attachments alongside the existing text/caption. After authorization and paused/offline checks, the runtime downloads each accepted image and injects a multimodal user message:

```ts
[
  { type: "text", text: captionOrFallback },
  { type: "image", data: base64, mimeType }
]
```

The fallback text for image-only prompts should be explicit, for example: `Please inspect the attached image.` Caption text remains the primary prompt when present.

Alternatives considered:
- Save Telegram images to local files and send `@file` references: rejected because extension-injected messages do not go through the interactive file-argument path, and it would leave extra local artifacts.
- Send a Telegram file link to the model: rejected because links may expire, require bot authorization, and leak the bot token if constructed carelessly.

### Authorize before download

Telegram update parsing records only safe file metadata (`file_id`, size, MIME/type hints, dimensions) before authorization. The runtime downloads bytes only after the update has passed the same binding, user, paused, online, and command-routing checks as text prompts.

Alternatives considered:
- Download during `getUpdates` parsing: simpler, but lets unauthorized chats consume bandwidth/storage and makes unsupported files harder to audit safely.

### Accept photos and image documents with conservative limits

For Telegram `photo`, choose the largest available size within the configured inbound size limit and infer `image/jpeg`. For Telegram `document`, accept only MIME types supported by Pi vision models in practice (`image/jpeg`, `image/png`, `image/webp`) and reject unsupported MIME types. Validate file size before and after download.

Add bounded image configuration with safe defaults, for example:

- `maxInboundImageBytes`
- `maxOutboundImageBytes`
- `maxLatestImages`
- `allowedImageMimeTypes`

Alternatives considered:
- Accept every Telegram document and let the model/provider fail: rejected because errors would be late and confusing.
- Add an image-processing dependency to resize large images: deferred to avoid increasing package footprint and complexity. Rejection with clear guidance is enough for v1.

### Respect model image capability

Before injecting image-bearing prompts, verify that the active Pi model advertises image input support (`model.input` includes `image`). If the current model does not support images, reject the multimodal prompt with a Telegram message explaining that the user should switch to a vision-capable model or resend text only.

Alternatives considered:
- Drop image blocks and send caption text only: rejected because it changes the user's intent silently.

### Extend route actions and broker IPC to carry multimodal content

Change `SessionRouteActions.sendUserMessage` from text-only to `string | (TextContent | ImageContent)[]`. In-process runtime calls it directly. Broker runtime adds image-aware payloads to the existing `deliverPrompt` request so the broker can download Telegram images and deliver multimodal content to the session-owning client over local IPC.

Alternatives considered:
- Have the broker write downloaded images to temp files and ask the client to read them: rejected because Pi's extension API already accepts base64 image content and temp-file coordination adds cleanup and security complexity.

### Track latest turn images in the session-owning extension

Collect image blocks from tool-result messages for the active turn and associate them with the assistant turn id at `agent_end`. Keep only a bounded latest-turn image list in memory, not in the Telegram binding metadata. Do not include local user image attachments in this list by default.

For the in-process runtime, `/images` can call `route.actions.getLatestImages()`. For the broker runtime, add a broker-to-client request such as `getLatestImages` so large image data is transferred only when the Telegram user explicitly asks for it, not during every route registration.

Alternatives considered:
- Store latest image base64 in `notification` and send it through every route sync: rejected because it can make normal broker synchronization large and slow.
- Persist latest images in custom session entries: rejected because tool-result/user messages already persist normal Pi image content, and tunnel metadata should remain small/non-secret.

### Explicit outbound image retrieval

Add `/images` and an optional inline `Download images` action on completion notifications when the latest turn has retrievable images. Sending images uses Telegram documents rather than photos so binary content and filenames are preserved and Telegram does not recompress the image. If no images are available, the command returns a clear fallback message.

Alternatives considered:
- Automatically send images on every completion: rejected for privacy, bandwidth, and chat-noise reasons.
- Send photos instead of documents: useful for preview UX but changes image bytes; documents are safer for artifacts and debugging screenshots.

### Make generated image files feel conversational

Telegram users naturally ask follow-up prompts like "send it as PNG" after Pi creates an image file. In practice, many image artifacts are created as workspace files by tools (`write`, renderers, converters, shell commands) and the assistant replies with a path instead of emitting a Pi image content block. The bridge should treat safe latest-turn image file references as explicit retrieval candidates, not only in-memory image content blocks.

The session-owning extension should collect candidate image file references from the latest turn by inspecting final assistant text and relevant tool-result text for obvious local image paths (`.png`, `.jpg`, `.jpeg`, `.webp`) in code spans, Markdown links, and plain text. Candidates are only staged as references. Bytes are read on demand when the authorized Telegram user invokes `/images`, taps the image button, or uses an explicit path command such as `/send-image <relative-path>`.

Validation must be fail-closed:

- accept only relative paths inside the current workspace/session root;
- reject absolute paths, `..` traversal, hidden state directories, and symlinks that resolve outside the workspace;
- verify file existence, regular-file status, MIME/magic bytes, allowed MIME type, and outbound size at retrieval time;
- deduplicate repeated references and cap candidate count with the same latest-image limit;
- preserve explicit retrieval: do not automatically send workspace image files just because a path appears in text.

The latest-image retrieval action can return both sources through the existing `LatestTurnImage` shape after loading bytes, while notification metadata can distinguish counts if useful (`contentImages`, `fileImages`, `skipped`). Completion messages should say "image output/file available" and expose the same inline button. If `/images` has no captured image blocks but the last assistant reply mentioned an image-looking path that failed validation, Telegram should explain the likely reason and suggest asking Pi to regenerate or read the image.

Alternatives considered:
- Automatically send any image path mentioned by Pi: rejected because it can exfiltrate local files and surprise users.
- Require the agent to always call `read` after creating an image: useful as a prompt/skill habit, but brittle and dependent on model behavior.
- Add a broad `/file <path>` browser: rejected; the feature should stay image-only and latest-turn/workspace-scoped.

## Risks / Trade-offs

- Large base64 payloads over broker IPC can be slow → enforce inbound size limits and only transfer outbound images on explicit request.
- Binary images can contain secrets that text redaction cannot remove → avoid auto-sending Pi-produced images and require explicit `/images` or button action.
- Current model may not support images → reject image-bearing prompts before injection with a clear message.
- Telegram MIME metadata can be missing or inaccurate → validate both declared MIME type and downloaded bytes where practical, and fail closed for ambiguous documents.
- Tool-result images may be numerous or large → bound count and per-image bytes, and report skipped images in Telegram.
- Latest-turn file references may accidentally mention sensitive or unrelated local images → require explicit retrieval, workspace containment, MIME validation, size limits, and latest-turn/path-command scoping.
- Broker and in-process runtimes can diverge → add shared test fixtures and broker IPC coverage for image prompt and image retrieval flows.

## Migration Plan

1. Add image-related configuration defaults while keeping existing configurations valid.
2. Extend types and Telegram API parsing in a backwards-compatible way; text-only updates continue to behave unchanged.
3. Implement in-process image prompt delivery and `/images` retrieval.
4. Add broker IPC changes and broker-runtime parity.
5. Add regression tests and manual smoke-test with a Telegram photo, image document, `/steer` caption, and latest image retrieval.
6. Rollback by disabling the package or reverting to a version before this change; no persistent migration is required beyond ignored config fields.

## Open Questions

- Should accepted image documents include `image/gif`, or should animated/ambiguous formats remain unsupported in v1?
- Should a future version add optional downscaling/transcoding instead of rejecting oversized image documents?
- Should outbound image retrieval include custom extension messages with image content, or only tool-result images from the latest agent turn?
